# Authentication Middleware (`authMiddleware.js`)

## Overview

The `authMiddleware.js` file is responsible for protecting secure routes in the application. It verifies whether incoming requests contain a valid authentication token before allowing access to protected resources.

This middleware acts as a security layer between client requests and restricted backend APIs.

---

# Responsibilities

The authentication middleware performs the following tasks:

- Extracts authentication token from request headers
- Validates the presence of the token
- Verifies JWT authenticity using a secret key
- Checks token expiration
- Attaches authenticated user information to the request object
- Prevents unauthorized access to protected routes
- Handles token-related errors securely

---

# Authentication Flow

## 1. Token Extraction

The middleware reads the token from the `Authorization` header.

Expected format:

```http
Authorization: Bearer YOUR_TOKEN
```

The `Bearer` prefix is removed before verification.

---

## 2. Token Validation

The middleware checks whether a token is present in the request.

If no token is provided:

- Access is denied
- A `401 Unauthorized` response is returned

---

## 3. JWT Verification

The token is verified using a secret key stored in environment variables.

Verification ensures:

- The token was generated by the server
- The token has not been tampered with
- The token is still valid

---

## 4. User Attachment

After successful verification:

- Decoded user information is attached to the request object
- This allows protected routes to access authenticated user data

Example usage inside routes:

```js
req.user
```

---

## 5. Access Control

If verification succeeds:

- The middleware calls `next()`
- Request processing continues to the protected route

If verification fails:

- Access is blocked
- Appropriate error response is returned

---

# Security Features

## JWT-Based Authentication

The middleware uses JWT (JSON Web Token) authentication to secure API routes.

---

## Token Expiration Handling

Expired tokens are detected automatically.

When a token expires:

- Access is denied
- User is asked to log in again

---

## Invalid Token Detection

The middleware detects:

- Malformed tokens
- Tampered tokens
- Incorrect signatures
- Invalid token formats

---

## Environment Variable Security

JWT secret keys are stored securely using environment variables instead of hardcoding them into source files.

---

# Error Handling

The middleware handles several authentication-related scenarios:

| Scenario | Response |
|---|---|
| No token provided | Unauthorized access |
| Expired token | Re-authentication required |
| Invalid token | Access denied |
| Verification failure | Invalid authentication |

---

# Dependencies Used

The middleware typically relies on:

| Dependency | Purpose |
|---|---|
| jsonwebtoken (JWT) | Token verification |
| Express.js | Middleware handling |
| dotenv | Environment variable management |

---

# Usage Example

Example of protecting a route using the middleware:

```js
router.get('/profile', authMiddleware, profileController);
```

In this example:

- The route becomes protected
- Only authenticated users with valid tokens can access it

---

# Expected Request Header

```http
Authorization: Bearer JWT_TOKEN
```

# Example Responses

## Unauthorized Access

```json
{
  "msg": "Access denied. No token provided."
}
```

---

## Expired Token

```json
{
  "msg": "Token expired. Please log in again."
}
```

---

## Invalid Token

```json
{
  "msg": "Invalid token."
}
```

# Purpose in Application Architecture

`authMiddleware.js` is a core part of the application's authentication and authorization system.

It is commonly used together with:

- Authentication Controllers
- Protected APIs
- User Management System
- JWT Authentication Flow
- Role-Based Access Control (Optional)

This middleware ensures that only authenticated users can access secure application resources.

---

# Notes

- All protected routes should use this middleware.
- JWT secret keys must remain private.
- Tokens should always be transmitted securely over HTTPS.
- Additional security enhancements such as refresh tokens and role validation can be added for production systems.
